## 2. Vulnerabilidades de SeguranĂ§a

### SEC #1 â Email de utilizadores exposto publicamente (ALTO)

`GET /api/users/{id}` retorna `UserProfileResponse` que inclui o campo `email`. Qualquer utilizador autenticado pode obter o email de qualquer outro utilizador por ID sequencial. PII leakage trivial.

### SEC #2 â Access tokens blacklistados pelo JWT completo, nĂŁo por JTI

Um JWT pode ter 400-600 bytes. A chave Redis Ă© `jwt_blacklist:<token_completo>`. Refresh tokens usam JTI (eficiente). Access tokens usam o string completo (ineficiente e inconsistente). Em alta carga com muitos logouts, as chaves Redis ficam extremamente grandes.

### SEC #3 â Rate limiting apenas por IP, sem persistĂȘncia

O rate limiter usa IP e vive em memĂłria. ContornĂĄvel com:
- MĂșltiplos IPs (atacante com botnet)
- Restart do container (reinicia contadores)
- Proxy reverso que mascara o IP real (o `X-Forwarded-For` nĂŁo Ă© lido)

### SEC #4 â `JwtService` expĂ”e mĂ©todos que deveriam ser privados

### SEC #5 â `Notification.payload` Ă© `String` com `columnDefinition = "jsonb"`

Hibernate vai tentar fazer `INSERT ... payload = '<string>'::text` mas a coluna Ă© `jsonb`. PostgreSQL vai rejeitar strings invĂĄlidas como JSON silenciosamente ou lanĂ§ar exceĂ§ĂŁo. O tipo correto seria `JsonNode` com `@JdbcTypeCode(SqlTypes.JSON)`.
