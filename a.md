# Auditoria Técnica — Code Arena (ft_transcendence)

---

## Resumo Geral: **6.1 / 10**

O projeto tem uma base sólida: arquitetura modular, JWT com rotação de refresh token, migrações Flyway, Docker com healthchecks. Mas tem features completamente não implementadas apresentadas como funcionais, N+1 queries, auth guard defeituoso, e um componente de perfil inteiro que é mock puro.

---

## 1. Bugs Críticos

### BUG #1 — `ProfileSettings` é mock completo (CRÍTICO)
profile-settings.ts

`saveAvatar()`, `savePersonalInfo()`, e `savePassword()` chamam `simulateDelay(800)` em vez de qualquer HTTP request. O componente usa `PLAYER_PROFILE_MOCK` como fonte de dados. **Nenhum dado é lido ou gravado na API.** Em produção, o utilizador pensa que atualizou o perfil, mas nada acontece.

```ts
// TODO: integrar com API (PUT /api/profile).
await this.simulateDelay(600);   // ← isto é o "backend" neste componente
```

### ~~BUG #2 — N+1 Query em `getMyMatches` (CRÍTICO)~~ ✅ RESOLVIDO
UserController.java

**Fix:** `UserService.getUsernamesByIds(Collection<Long>)` resolve todos os opponentIds num único `SELECT … WHERE id IN (…)`. O controller recolhe todos os IDs antes do stream e faz um só `getOrDefault()` por duel. 50 duels = **2 queries** em vez de 51.

### ~~BUG #3 — `authService.logout()` fire-and-forget no frontend~~ ✅ RESOLVIDO
auth.service.ts

**Fix:** `finalize(cleanup)` do RxJS garante que localStorage só é limpo (e o router navega para `/login`) **após** a resposta do backend — tanto em sucesso como em erro. O backend tem sempre a oportunidade de blacklistar os tokens antes da sessão local ser destruída.

### ~~BUG #4 — `isLoggedIn()` não valida expiração do JWT~~ ✅ RESOLVIDO
auth.service.ts

**Fix:** `isLoggedIn()` agora chama `isTokenExpired(token)`, que faz `atob()` do payload base64url do JWT e compara `decoded.exp * 1000` com `Date.now()`. Tokens expirados ou malformados retornam `false` e são removidos proativamente do `localStorage`. Não valida a assinatura (responsabilidade do backend) — é suficiente para o guard de UX.

### ~~BUG #5 — `Sidebar.ngOnInit` com condição estática em Signal reativo~~ ✅ RESOLVIDO
sidebar.ts

**Fix:** `ngOnInit` removido. Substituído por `effect()` registado no `constructor`. O `effect()` reavalia automaticamente sempre que o signal `isLobby` muda de valor — navegar para `/lobby` a partir de qualquer outra rota dispara `loadFriends()` correctamente. Com `ngOnInit`, a condição era avaliada uma única vez no arranque do componente, pelo que utilizadores que navegassem para `/lobby` depois de já estarem noutra rota nunca veriam a lista de amigos carregada.

### ~~BUG #6 — `Lobby.loadMatches()` sem error handler~~ ✅ RESOLVIDO
lobby.ts

**Fix:** Adicionado `matchHistoryError = signal(false)` ao `Lobby`. O `error` handler do subscribe ativa o signal; em sucesso é reposto a `false`. O `TerminalHistory` recebe o novo input opcional `errorLoading: input<boolean>(false)` e exibe "Failed to load match history. Please refresh the page." em vez do empty state genérico quando a flag está ativa. Falhas de rede, timeout ou 5xx deixam de ser silenciosas.

### BUG #7 — `LoginRateLimiter` tem memory leak gradual
LoginRateLimiter.java

O `attemptsCache` (ConcurrentHashMap) cresce indefinidamente. Entradas bloqueadas são removidas apenas quando o mesmo IP tenta de novo após o block expirar. IPs que nunca retornam ficam na memória para sempre. Em produção com tráfego variado, isto drena memória progressivamente.

### BUG #8 — Race condition no `addFriend`
UserProfileService.java

```java
if (!friendshipRepository.existsByUserIdAndFriendId(userId, friendId)) {
    friendshipRepository.save(...);   // janela de race condition aqui
}
```

Dois requests simultâneos passam no `existsBy...` e ambos tentam `save()`. O `@Transactional` na classe não resolve isto porque a verificação e o save são operações distintas sem lock de BD. Necessita de `INSERT ... ON CONFLICT DO NOTHING` ou constraint única explorada.

## 2. Vulnerabilidades de Segurança

### SEC #1 — Email de utilizadores exposto publicamente (ALTO)
UserController.java

`GET /api/users/{id}` retorna `UserProfileResponse` que inclui o campo `email`. Qualquer utilizador autenticado pode obter o email de qualquer outro utilizador por ID sequencial. PII leakage trivial.

```java
public record UserProfileResponse(
    Long id, String username, String email,  // ← email exposto em GET /api/users/{id}
```

### SEC #2 — Access tokens blacklistados pelo JWT completo, não por JTI
TokenBlacklistService.java

Um JWT pode ter 400-600 bytes. A chave Redis é `jwt_blacklist:<token_completo>`. Refresh tokens usam JTI (eficiente). Access tokens usam o string completo (ineficiente e inconsistente). Em alta carga com muitos logouts, as chaves Redis ficam extremamente grandes.

### SEC #3 — Rate limiting apenas por IP, sem persistência
AuthController.java

O rate limiter usa IP e vive em memória. Contornável com:
- Múltiplos IPs (atacante com botnet)
- Restart do container (reinicia contadores)
- Proxy reverso que mascara o IP real (o `X-Forwarded-For` não é lido)

O IP deveria ser `servletRequest.getHeader("X-Real-IP")` quando atrás de Nginx.

### SEC #4 — `JwtService` expõe métodos que deveriam ser privados
JwtService.java

`generateToken(Map, UserDetails, long, String)` é `public`. Qualquer código dentro do mesmo processo pode gerar tokens com lifetime e tipo arbitrários. Deve ser `private`.

`extractClaim` também é `public` — exposição desnecessária de internals.

### SEC #5 — `Notification.payload` é `String` com `columnDefinition = "jsonb"`
Notification.java

```java
@Column(nullable = false, columnDefinition = "jsonb")
private String payload;
```

Hibernate vai tentar fazer `INSERT ... payload = '<string>'::text` mas a coluna é `jsonb`. PostgreSQL vai rejeitar strings inválidas como JSON silenciosamente ou lançar exceção. O tipo correto seria `JsonNode` com `@JdbcTypeCode(SqlTypes.JSON)`.

### SEC #6 — `document.getElementById` no ProfileSettings
profile-settings.ts

```ts
const input = document.getElementById('avatar-input') as HTMLInputElement;
```

Anti-pattern Angular. Se dois instances do componente coexistirem, apanha o primeiro elemento com aquele ID na página. Sem segurança de tipo real. Deve usar `@ViewChild`.

## 3. Problemas de Performance

### PERF #1 — `recalculateLeagues()` a cada login de Master+
UserService.java

Cada login de um jogador com elo ≥ 3000 dispara um `UPDATE users SET league = CASE...` em **toda a tabela** de utilizadores com elo ≥ 3000. Com 10.000 utilizadores Master+, isto é um lock pesado na BD a cada login.

### PERF #2 — `enrichWithRankingContext` faz múltiplas queries por request `/me`
UserService.java

Para jogadores com elo ≥ 3000:
1. `countAllPlayers()` — COUNT(*)
2. `countPlayersWithEloAbove()` — COUNT com WHERE
3. `findEloAtGlobalRank()` — SELECT com OFFSET
4. `findHighestElo()` — MAX(elo)

4 queries síncronas por GET /api/users/me para qualquer jogador Master/Legend.

### PERF #3 — Leaderboard sem cache
LeaderboardController.java

Redis já está disponível no projeto. O leaderboard é público, a mesma query corre para cada visitante, e os dados mudam raramente. Sem `@Cacheable` ou TTL.

### PERF #4 — `getMyMatches` sem paginação
Retorna todos os duels do utilizador. Um jogador veterano com 500 duels recebe todos de uma vez.

## 4. Problemas de Arquitetura

### ARCH #1 — `UserController` viola camada de serviço
UserController.java

```java
private final DuelRepository duelRepository;          // repositório direto no controller
private final FriendshipRepository friendshipRepository;
```

Controllers não devem falar com repositórios. Toda a lógica de `getMyMatches` devia estar num `DuelService`. Isto quebra separação de responsabilidades e torna os testes unitários do controller impossíveis sem mockar JPA.

### ARCH #2 — Sistema de amigos inconsistente
O `FriendshipRepository` tem `findPendingForUserId` e a entidade suporta status "PENDING". O `addFriend` adiciona como "ACCEPTED" imediatamente. As queries de pending são dead code. O sistema implementado é "adicionar amigo instantaneamente sem aceitação" — pode ser intencional, mas as abstrações de "PENDING" são código morto confuso.

### ARCH #3 — `Notification` e `Message` entities sem implementação
Existem entidades JPA, tabelas na BD (via Flyway), mas zero controllers, services, ou endpoints. São placeholders que aumentam o schema e a complexidade da migração sem valor funcional atual.

### ARCH #4 — `QueuePanel` é UI fake
queue-panel.ts

O timer de queue funciona. Mas não há WebSocket, não há chamada à API, não há matchmaking real. O botão "entrar na fila" é decorativo. O status `IN_QUEUE` existe na entidade `User` mas nunca é ativado.

### ARCH #5 — Gestão de estado fragmentada no Lobby
O `Lobby` component busca os dados com `loadMe()` e `loadMatches()` em dois subscribes separados no `ngOnInit`. Não há coordenação entre as duas chamadas. Se `loadMe` falhar, `loadMatches` ainda corre. Se `loadMatches` falhar, é silencioso. Deveria usar `forkJoin` ou `combineLatest`.

## 5. Qualidade de Código

### QC #1 — `UserProfileResponse.from()` diverge de `leagueFromElo()`
UserProfileResponse.java

`from()` chama `leagueFromElo()` para o campo `league`. `leagueFromElo()` nunca retorna "LEGEND" (retorna "MASTER" para elo ≥ 3000). A lógica de LEGEND é depois aplicada via `withLegendContext()`. Este fluxo de duas etapas é frágil — se alguém chamar `from()` diretamente sem `enrichWithRankingContext`, jogadores Legend aparecem como Master.

### QC #2 — `Friendship.status` é String em vez de Enum
String literal "ACCEPTED" espalhada pelo código (`FRIENDSHIP_ACCEPTED = "ACCEPTED"`). Sujeito a typos. A entidade tem campos implicitamente relacionados com o enum `UserStatus` — o padrão deveria ser consistente.

### QC #3 — `LoginRequest` sem validação de comprimento
LoginRequest.java

`@NotBlank` mas sem `@Size`. Um payload com username de 10MB passa pela validação e vai até ao `authenticationManager.authenticate()`. Negação de serviço local via request body gigante.

### QC #4 — `UserProfile` interface no frontend tem `displayName` em falta
user-profile.model.ts

O backend retorna `displayName` no `UserProfileResponse`. O modelo TypeScript não o tem definido. O campo é perdido silenciosamente — tipo desalinhado entre contrato de API e cliente.

### QC #5 — `CommonModule` desnecessário em `ProfileSettings`
```ts
imports: [CommonModule, ReactiveFormsModule, RouterLink]
```
Angular 17+ com standalone components não precisa de `CommonModule`. `*ngIf` e `*ngFor` são substituídos por `@if` e `@for`. O componente usa templates com diretivas `@if`, por isso o `CommonModule` é importado mas não usado.

## 6. O que está bem feito

- **JWT com refresh token rotation + Redis blacklist**: implementação correta e segura. A revogação por JTI em refresh tokens é o padrão da indústria.
- **BCrypt com fator 12**: parâmetro de segurança adequado.
- **Path traversal protection no upload de avatar**: `targetPath.startsWith(storagePath)` correto.
- **Bean Validation nos DTOs** com mensagens de erro claras.
- **Flyway** para migrações — BD nunca é gerida por `ddl-auto=update`.
- **Docker Compose com healthchecks** e `depends_on: condition: service_healthy` — serviços só sobem quando as dependências estão prontas.
- **Lazy loading de rotas** no Angular — bundle inicial mínimo.
- **Signals do Angular** usados de forma idiomática onde existem.
- **CORS configurável** via environment variables.
- **STATELESS sessions** — sem CSRF, sem sticky sessions.
- **`QueuePanel.ngOnDestroy`** limpa o `setInterval` corretamente.
- **`JwtAuthenticationFilter`** não faz short-circuit em falhas — endpoints públicos continuam acessíveis.

## Nota Final — Perspectiva de Tech Lead

**6.1/10**

O esqueleto do projeto é competente: stack moderna, segurança JWT bem pensada, infra Docker funcional, separação de módulos clara. Quem escreveu isto percebe de Spring Boot e Angular.

Mas o projeto tem o problema clássico de aparecer mais completo do que está: features visíveis na UI que são completamente mock (`ProfileSettings`), componentes com botões funcionais que não fazem nada (`QueuePanel`), entidades no schema que não têm endpoints. Isto é tecnicamente dívida técnica mascarada de feature completa.

Os problemas mais graves não são de complexidade — são de rigor: um `isLoggedIn()` que ignora expiração, um logout fire-and-forget, um N+1 óbvio, e PII leakage num endpoint público. Estes são erros que passam em code review quando não há testes de integração e não se verifica o comportamento real em produção.

Para chegar a 8+/10, o foco deve ser: fechar o gap entre o que a UI mostra e o que a API suporta, adicionar testes (há uma pasta de testes mas não explorei cobertura — suspeito que é baixa), e eliminar os problemas de N+1 e segurança listados acima.
